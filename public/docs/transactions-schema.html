Transactions schema

Six new Postgres tables that capture the transactions feature: a typed taxonomy of categories/sub-categories, a rule engine for auto-categorization, and a three-level transaction model that handles a single bill spanning multiple properties with multiple owners each.

The three-level model

transaction                          // totals: $1000 utilities, type=expense
  ├─ transaction_property_split      // Property A — 60% — $600
  │    ├─ transaction_allocation     // Owner X — 50% — $300
  │    └─ transaction_allocation     // Owner Y — 50% — $300
  └─ transaction_property_split      // Property B — 40% — $400
       └─ transaction_allocation     // Owner X — 100% — $400

transaction root
One row per recorded financial event — totals only

No property_id column. Every transaction reaches its property(ies) through transaction_property_split — even single-property transactions have one split row.

ColumnTypeNotes
PK idUUIDDEFAULT gen_random_uuid()
FK org_idUUID→ organisation
FK entity_idUUID→ entity (CASCADE)
NN typeVARCHAR(16)CHECK in (revenue, expense)
FK category_idBIGINT→ transaction_category
FK subcategory_idBIGINT→ transaction_subcategory
NN invoice_dateDATE—
NN gross_amountNUMERIC(14,2)CHECK ≥ 0. TOTAL across splits.
NN gst_amountNUMERIC(14,2)CHECK ≥ 0. DEFAULT 0.
NN net_amountNUMERIC(14,2)CHECK = gross − gst
descriptionTEXT—
internal_remarksTEXT—
is_asset_purchaseBOOLDEFAULT false. Capital purchase flag (expense only).
asset_classVARCHAR(20) NULLCHECK in (capital_works, capital_allowance). Required when asset purchase.
effective_life_yearsNUMERIC(5,1) NULLCHECK > 0. Required only for capital_allowance.
FK rule_idBIGINT NULL→ transaction_rule (SET NULL)
review_statusVARCHAR(16)DEFAULT 'unreviewed'. Rules apply only to unreviewed.
metadataJSONBDEFAULT '{}'. CHECK object.
created_by / updated_byVARCHAR(10)→ users
is_deletedBOOLDEFAULT false

Asset-purchase CHECK: when is_asset_purchase = true, type must be 'expense' and asset_class must be set; when asset_class = 'capital_allowance', effective_life_years must be > 0; otherwise all three are NULL/false.

transaction_property_split level 2
Per-property slice of one transaction

For a single-property transaction: exactly one row at split_percentage = 100. For a multi-property bill: N rows summing to 100% (handler-enforced). UNIQUE on (transaction_id, property_id).

transaction_allocation level 3
Per-owner share of one property split (snapshotted)

Snapshots the owner’s percentage at insert time. owner_name + entity_beneficiary_id are denormalized so audit data survives if property_owner rows are deleted later.

Example C Rule creation — auto-categorize Origin Energy as Utilities → Electricity 1 row in transaction_rule

Create Rule

If — match ALL of
DescriptionContains“Origin Energy”
AmountIs greater than50
Then assign
TypeExpense
CategoryUtilities
Sub-categoryElectricity
Auto-confirmoff

Single insert — conditions array embedded as JSONB

INSERT INTO transaction_rule
  (org_id, entity_id, property_id,
   name, match_mode, conditions,
   assigned_type, assigned_category_id, assigned_subcategory_id,
   auto_confirm, is_enabled, created_by)
VALUES
  ('…org…', '…entity…', NULL,  -- All Properties
   'Origin Energy → Utilities', 'all',
   '[
      {"field":"description","operator":"contains","value":"Origin Energy"},
      {"field":"amount","operator":"gt","value":50}
    ]'::jsonb,
   'expense', 87, 202,
   false, true, 'usr_8a3c');

Later — when a matching transaction is reviewed, the rule’s id is recorded on it

UPDATE transaction
  SET rule_id = 3014,
      review_status = 'reviewed',
      category_id = 87,
      subcategory_id = 202,
      updated_by = 'usr_8a3c'
WHERE id = 'txn_e7f8g9'
  AND review_status = 'unreviewed';

Counter-examples — what the CHECK rejects

-- ❌ asset purchase on a revenue row
INSERT INTO transaction (..., type, is_asset_purchase, asset_class, ...)
VALUES (..., 'revenue', true, 'capital_works', ...);
-- ERROR:  new row violates check constraint
--         "transaction_asset_purchase_check"

-- ❌ capital_allowance without effective_life_years
INSERT INTO transaction (..., is_asset_purchase, asset_class, effective_life_years, ...)
VALUES (..., true, 'capital_allowance', NULL, ...);
-- ERROR:  new row violates check constraint
--         "transaction_asset_purchase_check"

-- ✅ capital_works can omit effective_life_years
INSERT INTO transaction (..., is_asset_purchase, asset_class, effective_life_years, ...)
VALUES (..., true, 'capital_works', NULL, ...);
-- OK
